 |指令名|功能描述|

 |-|-|

 |DMB|数据存储屏障（Data Memory Barrier）。DMB指令保证：仅当所有在它前面的存储器访问操作都执行完毕后，才提交(commit)在它后面的存取访问操作指令。当位于此指令前的所有内存访问均完成时，DMB指令才会完成|

 |DSB|数据同步屏障（Data synchronization Barrier）。比DMB要严格一些，仅当所有在它前面的存储访问操作指令都执行完毕后，才会执行在它后面的指令，即任何指令都要等待DSB前面的存储访问完成。位于此指令前的所有缓存，如分支预测和TLB维护操作全部完成|

 |ISB|指令同步屏障（Instruction synchronization Barrier）。它最严格，冲洗流水线(Flush Pipeline)和预取buffer(prefetch buffers)后，才会从cache或者内存中预取ISB指令之后的指令。ISB通常用来保证上下文切换的效果，例如更改ASID(Address Space Identifier)、TLB维护操作和C15寄存器的修改等。|

semaphore 信号量

  • 信号量允许计数值大于1，也就是说允许多个进程进入临界区访问共享资源。当计数值等于1时，只允许一个进程进入，此时信号量类似于互斥锁
  • 信号量不允许在中断处理中使用，因为任务获取失败后会被推入等待队列，进入不可中断的睡眠状态，而中断不允许睡眠
  • 由于争用信号量的进程在等待锁重新变为可用时会睡眠，所以信号量适用于锁会被长期持有的情况
  • 信号量可以由一个线程释放，另一个线程得到，所以信号量可以用于线程的同步

 3.PV操作

 信号量支持两个原子操作P( )和V( )，前者是测试操作，而后者是增加操作。Linux把这两种操作分别成为down( )和up( )。

 down( )操作通过对信号量计数减1来请求获得一个信号量，如果结果是0或者大于0，获得信号量锁，任务就可以进入临界区。如果结果是负数，任务会被放入等待队列，处理器执行其它任务。与之相反，当临界区操作完成后，up( )操作会释放信号量，增加信号量的计数值。如果在该信号量上的等待队列不为空，则处于队列中等待的任务在被唤醒的同时会获得该信号量。

Spinlock 自旋锁

  • 自旋锁一般用在临界区代码非常少的情况（如果执行时间过长，在锁外面忙等的CPU比较浪费，特别是自旋锁临界区不能睡眠）
  • 自旋锁采用busy-wait-loop
  • 自旋锁不允许睡眠（否则会出现死锁），无法获得锁时就会自旋等待
  • 拥有自旋锁的临界区代码必须原子地执行，不能休眠和主动调度

 在普通自旋锁中，如果在临界区内发生了外部硬件中断，则系统会暂停当前进程的执行而处理该中断。在这种情况下，如果中断处理机制也需要对临界区进行操作，那么在操作之前会调用spinlock函数来保护该临界区。此时，由于中断处理程序会试图获取该自旋锁，而该自旋锁往往已经被其它CPU持有，故中断处理程序会进入忙等状态或者wfe睡眠状态。在这种情况下，自旋锁的持有者因为被中断打断而不能尽快释放锁，而中断处理程序一直在忙等该锁，从而会产生死锁。为了解决这一问题，Linux内核引入了自旋锁的变体spin_lock_irq()：

 spin_lock_irqsave的功能是保存本地CPU当前的中断状态并且关闭本地CPU中断，然后获取自旋锁。这样做的好处在于，如果在外部硬件中断发生之前本地CPU已经关闭中断，那么如果调用了spin_lock_irq，本地CPU会重新允许中断，这样的情况显然是不被允许的。而如果调用了spin_lock_irqsave，那么在进行相关中断开启/关闭之前首先会检查是否允许中断(flags)，这样就避免了重新开启本地CPU中断情况的发生。

 3.MCS lock

 现在我们把目光转向MCS锁。前文我们介绍spinlock已经有了较高的性能和较好的可拓展性，且基于ticketlock 实现的spinlock也解决了公平性的问题，那为什么我们还需要引入MCS锁呢？其实，spinlock有一个较为严重的问题，就是它是在一个变量上自旋的，这就导致在多处理器系统中，每个试图获取该锁的进程都将把该变量所在位置的缓存行读入本地CPU的cache中，使得该cache行进行频繁的换入换出操作，效率大幅下降。且每次的读写操作都必须在多个处理器缓存之间进行缓存同步，导致繁重的系统总线和内存的流量，大大降低系统整体的性能。

 为了解决这个问题，内核社区的专家们设计出了MCS算法。MCS算法的最大不同点在于锁的申请者在本地的CPU变量上自旋，而不是在全局变量上。OSQ锁(optimistic_spin_lock)是MCS锁的一种具体实现，我们现在来看看它的实现机制。

 但是，在进程自旋等待的过程中，如果该进程被调度器调度出去了(need_resched())或者被更高优先级的进程抢占(vcpu_is_preempted())，此时将跳转到unqueue标签中，以执行退出MCS链表、放弃自旋等待操作，此即慢速申请通道。

 为了从MCS链表中删除结点，我们首先解除前一结点的next指针的指向。如果目前前继结点的next指针仍然指向当前结点，则说明在此期间链表还没有被修改，此时直接原子地把前继结点的next指针指向NULL即完成了第一步操作；如果前继结点的next指针不指向当前结点，则说明链表此时已经发生了变化，这里的变化可能有两种，一种是前继结点持有锁后释放，此时锁将传递给当前结点，这时的smp_load_acquire()函数将判断是否发生了这种情况，如果是的话将直接返回true。另一种情况则较隐蔽一些，其实可能这时前继结点也因为调度或抢占等原因离开了链表，所以此时要重新读入prev指针，并再度进入循环，重复这些操作。

Mutex 互斥锁

  • Mutex是在原子操作API基础上实现信号量的行为
  • 互斥锁是一种休眠锁，锁争用时可能存在进程的睡眠和唤醒，context切换代价较高，适用于加锁时间较长的场景
  • 互斥锁每次只允许一个进程进入临界区
  • 互斥锁的加锁和解锁必须在同一线程里对应使用，所以互斥锁只能用于线程的互斥

互斥锁实现

 1.互斥锁结构

 atomic_long_t->int count，所描述的owner为原子计数，当owner为0时表示锁没有被持有，反之则锁被持有；wait_list为等待队列；wait_lock则为自旋锁，用以保护wait_list。

 在mutex中，owner可以被划分为两个域：63bit至3bit存放锁的持有者task_struct地址，而2bit至0bit存放锁的状态：bit0表示wait_list是否为空，bit1表示释放锁的时候需要把锁交给第一等待者，而bit2表示handoff已经完成，等待task域来占用锁。

乐观自旋：

举例，不带乐观自旋的 mutex 版本中，若 A 线程持有锁且正在运行，此时 B 线程想拿锁，发现锁正被 A 拿着，B 将立即进入睡眠状态，在 A 线程释放锁后再去唤醒 B 线程来拿锁。但是我们通常认为持有锁的 A 若是在运行的情况下，通常会很快的退出临界区并释放锁，那么 B 去睡眠则是不必要的，毕竟睡眠和唤醒也是有代价的，完全可以多等一会即可。那么当 mutex 开启乐观自旋的 feature 后，若 B 拿不到锁，且发现锁的持有者 A 仍占用着 CPU 运行时，则不再去睡眠，而是像自旋锁一样进行自旋等待，直到 A 释放锁，但期间若 A 失去 CPU（即p->on_cpu为0），那么 B 将不会继续傻傻的自旋等待，而是进入睡眠。

唤醒睡眠线程可能需要很长时间，一旦该线程开始运行，它可能会发现处理器缓存不包含任何数据，从而导致过时的缓存未命中。相反，spin 等待 mutex 的线程将能够在它可用时快速获取锁，并且可能仍然是 cache hot 的。很显然，乐观自旋机制能提高 mutex 和整体的性能。

handoff机制：

当 mutex 支持乐观自旋时，那么会存在这样的情况：wait_list中有一些进程在睡眠并等待被唤醒拿锁，同时还有一些进程不在wait_list中且不断的自旋等锁或乐观自旋等锁。wait_list上的进程大概率是抢不过自旋拿锁的进程的，这是因为调度延时的存在。当被唤醒的进程真正获得 CPU 时，锁早就被自旋等锁的进程给偷走了(偷锁)，这会在一定程度的造成wait_list的等待者长时间“饥饿”。
